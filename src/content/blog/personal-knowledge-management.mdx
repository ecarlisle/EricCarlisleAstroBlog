After a decade of trying every note-taking app, I settled on plain Markdown files in a git repository. It's boring, portable, and searchable with tools that will outlive every SaaS product.

## The Structure

```
~/notes/
  /projects/          # One file per project
  /concepts/          # One file per technology or idea
  /log/               # Daily work logs
    2024/
      01-jan.md
      02-feb.md
  /references/        # Book notes, conference talks
  /dotfiles/          # Config snippets I always need
```

No nesting deeper than two levels. Deep hierarchies are where notes go to die.

## Linking Notes

I use `[[wiki-style links]]` within files — a simple grep finds all references to a concept across every note. No database needed. This is the most important habit: if a note mentions something that has its own file, link it.

## Daily Logs

Each workday gets a file. I timestamp significant events — a build that took too long, a bug that took three hours to find, a library that surprised me. These are not journal entries; they're evidence. When I encounter a similar problem six months later, I grep my logs.

## Search

`rg` (ripgrep) searches the entire repository in under a second. I alias `n` to search my notes directory:

```bash
alias n='rg --no-heading --smart-case ~/notes'
```

If I can't find it in three searches, I write a new note with better keywords.

## The Rule

If I've solved a problem twice, I write it down. The third time, I search before I start. That's the threshold where the system pays for itself.

Design tokens are the atoms of a design system. When they're well-structured, changing the global border radius or primary color is a one-line PR. When they're not, you're hunting through CSS files at 2 AM.

## Naming Conventions That Scale

The most durable naming system follows a category-concept-modifier pattern:

```
--color-surface-primary
--color-surface-elevated
--space-inline-sm
--space-inline-lg
```

Avoid naming by visual effect (`--color-shadow-light`) or by use case (`--sidebar-bg`). The first leaks implementation details and the second creates a new token every time a component needs its own color.

## Distribution

For a single project, CSS custom properties in a central file work fine. For multi-product systems, you need a package:

```
@mycompany/tokens
  /tokens
    color.json
    spacing.json
    typography.json
  /dist
    tokens.css
    tokens.esm.js
```

Build the dist from source JSON so design tools and developers consume the same values. Style Dictionary handles this transformation reliably.

## Versioning

Tokens are a public API. Breaking changes — renaming a token, removing one, changing its value in a way that breaks a component's visual contract — require a major version bump. Additive changes are minor. Documentation-only changes are patch.

This discipline forces teams to think before renaming things, which is precisely the friction you want.

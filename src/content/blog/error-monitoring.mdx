Error monitoring tells you what broke for real users in production. Without it, you're debugging blind — relying on users to file bug reports that usually say "it didn't work" with no reproduction steps.

## Choosing a Service

Sentry is the standard for frontend error tracking. It groups errors intelligently, captures context (browser, OS, URL, previous actions), and supports source maps for readable stack traces. OpenReplay adds session replays if you need to see what the user was doing before the error occurred.

## Source Maps Are Non-Negotiable

Minified stack traces are useless. Upload source maps to your error tracker as part of your build pipeline:

```bash
sentry-cli releases files $VERSION upload-sourcemaps ./dist
```

Make sure source maps are not publicly accessible — Sentry uses authentication, but publicly hosted source maps expose your source code to anyone who inspects the network tab.

## Breadcrumbs for Context

Configure breadcrumbs to capture meaningful user actions before the error:

```js
Sentry.addBreadcrumb({
  category: 'navigation',
  message: 'Navigated to /settings',
  level: 'info',
});
```

When an error occurs, the breadcrumb trail tells you what the user was doing, which is usually more useful than the error message itself.

## Noise Reduction

Not every error needs a notification. Ignore known non-issues — `ResizeObserver loop limit exceeded`, `Script error` (cross-origin), and errors from browser extensions. Set a reasonable rate limit for the same error to avoid alert fatigue.

Aim for a signal-to-noise ratio where every alert prompts action. If you're ignoring more than you're reading, tighten your filters.

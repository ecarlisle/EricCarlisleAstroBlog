React Server Components (RSC) are often explained as "components that run on the server," which is technically true but misses the point. SSR also runs components on the server. The difference is in the output: SSR sends HTML, RSC sends a serialized description of the component tree that the client can merge with its existing state.

## What Makes RSC Different

A Server Component can access the database directly, read from the filesystem, or call internal APIs — and none of that code ever reaches the client bundle.

```tsx
// This component runs only on the server.
// The SQL query, the data transformation, and the imports never ship to the browser.
async function RecentPosts() {
  const posts = await db.query('SELECT * FROM posts ORDER BY created_at DESC LIMIT 5');
  return (
    <ul>
      {posts.map(post => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  );
}
```

## The Client Boundary

The rule is simple: Server Components can import Client Components, but not the reverse. Add `'use client'` at the top of any file that needs interactivity, event handlers, or hooks like `useState` and `useEffect`.

This boundary is where most confusion lives. The solution is to push interactivity as far down the tree as possible, wrapping only the interactive parts in `'use client'` while keeping the rest server-rendered.

## When to Use RSC

RSC excels for data-heavy pages where most of the UI is static. Dashboards, product listing pages, and content-heavy sites benefit most. Highly interactive applications with constant state changes gain less.

## Performance Impact

The server component payload is more compact than the equivalent HTML because it's a binary format. Combined with streaming, this means the page starts rendering before all data is available — a meaningful improvement for slow database queries.

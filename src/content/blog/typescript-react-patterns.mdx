TypeScript with React can be either a joy or a constant fight. The difference usually comes down to a few patterns that handle the awkward edges well.

## Generic Components

When a component accepts a value and an onChange, infer the type from the props:

```tsx
type Option<T extends string> = { label: string; value: T };

function Select<T extends string>({ options, value, onChange }: {
  options: Option<T>[];
  value: T;
  onChange: (value: T) => void;
}) {
  return (
    <select value={value} onChange={e => onChange(e.target.value as T)}>
      {options.map(o => (
        <option key={o.value} value={o.value}>{o.label}</option>
      ))}
    </select>
  );
}
```

Now `onChange` receives the exact union type of your options, not a generic `string`.

## Discriminated Union Props

Use a `type` discriminant to enforce mutually exclusive prop combinations:

```tsx
type ButtonProps = {
  label: string;
} & (
  | { variant: 'primary'; action: () => void }
  | { variant: 'link'; href: string; external?: boolean }
);
```

TypeScript enforces that you provide `action` with primary and `href` with link. Impossible states are impossible to represent.

## When Not to Use forwardRef

If your component is a simple wrapper around a native element, use `forwardRef` so parent components can access the DOM node. If your component manages internal state and doesn't expose a DOM-facing API, skip it. The ergonomic cost of `forwardRef` with generic types is higher than most people admit.

Starting a component library is deceptively hard. The first few components are easy. The 50th — when you're dealing with composition, theming, and cross-team API preferences — is where most libraries fall apart.

## Framework Choice

If your organization uses one framework, build for it. If it uses multiple, build with web components or use a framework-agnostic approach with a thin adapter layer. Radix UI and Ariakit provide headless primitives that you style, which is usually a better investment than building from zero.

## API Design Principles

- Minimize required props. If a good default exists, use it.
- Spread remaining props onto the root DOM element so consumers can override anything.
- Expose `asChild` polymorphic patterns for layout flexibility.

```tsx
type Props = React.ComponentPropsWithoutRef<'button'> & {
  variant?: 'primary' | 'secondary';
  loading?: boolean;
};

function Button({ variant = 'primary', loading, children, ...props }: Props) {
  return (
    <button className={`btn btn--${variant}`} disabled={loading} {...props}>
      {loading ? <Spinner /> : children}
    </button>
  );
}
```

## Documentation

Storybook is the standard for a reason. Every component should have stories for its default state, every variant, error state, loading state, and edge case. If it's not documented, it might as well not exist.

## Testing

Test behavior, not implementation. A button test should verify that `onClick` fires, not that a specific CSS class is present. Use Testing Library for React components and Playwright for integration tests that verify real browser behavior.
